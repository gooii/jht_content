Development roadmap

Historical Texts is in continual development. You can see the list of features we've scheduled or are currently working on below, along with the ones we've already released.

In development

When work is scheduled we'll add it to the roadmap with the "scheduled" status. When we start work on a feature the status will change to "in development". It will then change to "in testing" when it's nearly ready for release and it's being tested internally. It's difficult to specify timescales for development work as we don't know what issues we'll encounter and need to overcome along the way though.

User evaluation of search and results pages and another look at the viewer

Mar - Apr 2014

During late March and early April the third round of user evaluation was conducted, looking at the search and results pages as well as another review of the book viewer.

The same Webinar format as previously was followed, with two sessions each week for the first two weeks, then a single session on the final week for the book viewer. A shared Google document was available for people to add comments to after the sessions after they had had time to familiarise themselves with the interface. Members of the Advisory Board and User Group were invited to attend and given access to the interface for testing purposes.

Browsing

The browse feature has been designed and initially shows some of the Thomason Tracts, so we asked if the display and navigation, as shown in the screenshot below, was as people would expect.

Searching

On the search page, we created an ‘add another’ row function to enable people to build up complex searches, so we asked how intuitive that is to use. We also explained that building Boolean searches using this method is difficult as there is no way to group terms by brackets, so you can’t tell the difference between:

(term1 AND term2) OR term3
term1 AND (term2 OR term3)

Although you can enter Boolean search terms with brackets directly into the search box, the search software provides an alternative by using MUST, SHOULD and MUST NOT instead. It does this by treating each term as ‘compulsory’ (it MUST appear in the results) or ‘optional’ (it SHOULD appear in the results). Using this method, the results would show everything that matched the ‘MUST’ field, with those that matched the ‘SHOULD’ field as well ranked higher. So if you were looking for all texts by Shakespeare, and wanted to see those with Venus in the title listed at the top of the results you would enter:

Author > Shakespeare

SHOULD contain > Title > Venus

As shown in the screenshot below.

Variant spellings (e.g. Shakespeare, Shakespear), printing substitutions (‘vv’ used instead of ‘w’) and other text variations (e.g. OCR errors such as f’ for the ‘long s’) were discussed, which would be the highest priority to support and whether printer substitutions need to be separated from general OCR errors.

Results display

We asked about the results list, whether the right information was displayed on it, whether it was intuitive to open a text using the thumbnail or title as well as opening a additional text in a new tab by right clicking the title on the results page. We also requested feedback on whether there was any other bibliographic information users would like the see under ‘more information’ for each result.

We asked whether it was clear how to get back to the results list from the viewer by using the ‘Results’ link near the top left as well as whether the navigation bar at the top needed any options adding or removing and whether it should appear when in the viewer as well.

The system currently provides live updates on the number of results as you type, so we asked if this was liked or whether people would want the update to only happen after you pressed the ‘search’ button. Questions were raised about what happens if you mistype and some people would expect to only see the number updated after they pressed ‘search’.

We also asked about the number of results per page, and whether people would want infinite scrolling, so more results would load as you scrolled down the list, or whether they wanted pagination, which is currently set to 100, 500, 1000, 2000 results per page. The preference seemed to be for pagination and there were discussions over the length of time it would take for larger numbers of results to be displayed, and whether it might make sense to be able to switch the thumbnails next to each result off in these cases so the results load faster.

What's next?

As with the previous rounds of consultation, the sessions were found extremely useful by the development team in terms of understanding how people would expect the system to work and what they want to be able to do with it. The feedback from these sessions has been incorporated into the development and we are now preparing to release the BETA version of Historical Texts in early May. All institutions who currently subscribe to Jisc Historic Books will be able to access and test the BETA version.

User evaluation of book viewer

Jan - Feb 2014

During late January and February the second round of user evaluation was conducted, this time taking a first look at some real software, rather than just designs.

The book viewer has been built first, as it’s using standard software and so could be created faster. It has three panels, the central one displaying the page image itself, then a panel to the left enabling you to navigate and search within the text using search terms and thumbnails, as well as navigating and searching your whole results set. A panel to the right will display the bibliographic information for the title you are looking at, the table of contents (where available), the text supporting the image (either the machine generated optical character recognition (OCR) for ECCO and BL, or manually keyed text for EEBO where available) and HTML where available.

The panels can be re-sized as required by dragging them to your desired position, and your settings can be remembered for the next time you use the system.

The central panel enables you to zoom in and out on the page image, expand or hide the panels to the left and right, switch to full screen mode, toggle whether highlighting is on or off and rotate the image.

Like with the first round of user consultation, over a period of three weeks, webinar sessions were run using WebEx to demo the book viewer and it’s features and ask for feedback. Our Advisory Board and User group were invited to attend the sessions facilitated by Sero Consulting, Owen Stephens Consulting, Gooii and the project manager Scott Gibbens at Jisc Collections, and, again, two sessions were run each week. Following each weeks feedback, changes were made accordingly to the book viewer, and the changes shown at the next iteration the following week.

Many interesting discussions were had during these sessions and some of the key topics covered included:

Terminology

This is something that has caused confusion in the current system, and we’re keen to try and make this as clear as possible in the new platform. There can be confusion between the terms 'image' number (number of the digitised image) and 'page' number (physical page in the printed text). There is also possible confusion with the term ‘image’ itself, which could mean 'page image' or 'illustration'. The terms ‘text’ (referring to the manually keyed EEBO text and the machine generated optical character recognition (OCR) for ECCO and the BL collections) and ‘html’ were also raised as not necessarily that clear, or the difference between them.

Zoom

The smooth zoom controls were liked, and the best way to indicate where you are on an image, once you have zoomed in, was discussed and a mini-page image was suggested.

Icons and Chevrons

Some people prefer icons, others prefer text to describe options in the viewer and where icons are used a tooltip was wanted when you hover over it, to make it clear what it does. Using chevrons to indicate the direction the left and right panels open in also caused some confusion, as some people thought they should point one way and others the opposite way.

Full screen

The option to remove all other information from the screen apart from the image viewed was offered and support was received for this feature.

Default layout

The new book viewer contains a lot of features, and not everyone will want to use all of them, or use them every time they view a text, so we asked what the default view should display when you first open a text. We offered an option with the page image in the central panel maximised, with the panels to the right and left minimised, so you can open them if and when you want to, and this was welcomed.

Navigation

Navigating by page number as well as image number was something people were really keen to have, with the appreciation that there can be inaccuracies in the pagination, and there aren’t always page numbers for all pages. As well as being able to locate all the illustrations in a text.

Double page images

The option to view two page images side by side was shown. For EEBO, which has been digitised by opening, this would mean being able to display four physical pages at a time. The feature was welcomed and discussions centered around whether you would want to navigate two images or one image at a time, which image you were on when zooming or displaying the supporting manually keyed or OCR text.

Thumbnails

The size of the thumbnails was discussed, and whether they should be available in different sizes, as well as whether it would be helpful to have image and/or page numbers on them.

What's next?

We're working on the search, browse, results display and homepage parts of the user interface now and the next round of user consultation in March will show real searches being run and results displayed. Further data analysis and data ingest is going on and images are being processed to provide the zoom feature.

In the meantime, if you have any comments, questions or feedback on the book viewer please do let the team know by contacting our helpdesk at historicbooks@mimas.ac.uk.

 

User evaluation of initial interface designs

Nov - Dec 2013

During November and December 2013 feedback was sought from the Jisc Historic Books Advisory Board and User Group on the initial designs for the new user interface created by our development partners Gooii. Essentially these are like screenshots of what each page in the interface could look like. At this stage it's easy to completely change the layout and options as no software has been programmed yet, but people can visualise how it will work and suggest alterations.

Over a period of three weeks, webinar sessions were run using WebEx to introduce the designs and discuss the features that will be available initially and the Advisory Board and User Group were invited to attend as many sessions as they could. The webinars were facilitated by our development partners at Sero Consulting, Owen Stephens Consulting, Gooii and the project manager Scott Gibbens at Jisc Collections. Each week two sessions were held to run through the same designs, on different days and at different times, to provide more flexibility for people to join one. Each week a new iteration of the designs, called a 'wireframe', was produced based on feedback from the previous week.

Members of the Advisory Board and User Group were also given the opportunity to feedback after the webinars through a Google User Evaluation document, which provided links to each individual wireframe and invited comments on them. Librarians at subscribing institutions were also given the opportunity to comment and pass details of the evaluation document on to interested academics and students at their institutions.

The design team at Gooii created the screencast below to provide a walkthrough after the final session in week three (select the full screen button in the bottom right hand corner of the video to maximise the viewing window).


Screencast created by Gooii

  • Basic search

  • Advanced search templates

These sessions were extremely useful for the development team to understand how the community want to use the system and feedback from the participants indicated they appreciated the opportunity to help shape the new platform.

Behind the scenes, the teams at Knowledge Integration and Mimas and our metadata expert at Owen Stephens Consulting, are busy analysing the data, creating import and processing frameworks, data models and data importers for each collection.

What's next?

Work has started to build the user interface, and the next round of user evaluation in late January and early February 2014 will be looking at real software rather than wireframes.

In the meantime, if you have any comments, questions or feedback on these designs please do let the team know by contacting our helpdesk at historicbooks@mimas.ac.uk.
